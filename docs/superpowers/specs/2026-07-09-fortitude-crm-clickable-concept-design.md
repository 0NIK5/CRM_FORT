# Fortitude CRM — кликабельный концепт (frontend-only). Дизайн

Дата: 2026-07-09
Статус: одобрено пользователем
Источник контекста: `docs/PROJECT_CONCEPT_Fortitude_CRM.md`

## 1. Цель

Собрать **кликабельный концепт** (не MVP, без рабочей логики) двух приложений
платформы «Фортітьюд ЮА»:

- **Admin CRM** — внутренний инструмент команды фонда
- **Donor Portal** — кабинет донора (read-only)

Задача — показать человеку, **как это будет выглядеть**. Требования:

- страницы кликабельны, переходы работают;
- **без аутентификации** — «вход» это просто переход по ссылке;
- **без базы данных и бэкенда** — данные статические (моки), правдоподобные, для вида;
- **frontend only** на этом этапе; бэк и интеграции — позже;
- результат задеплоен через **GitHub Actions на GitHub Pages**, чтобы можно было
  отправить публичную ссылку.

Явно вне охвата (YAGNI на этом этапе): реальный логин, API, БД, row-level
security, интеграции 1С/Kobo, формы с сохранением, серверная валидация.

## 2. Стек и архитектура

- **Next.js (App Router) + static export** (`output: 'export'`) + **Tailwind CSS**.
  На выходе — статика в `out/`, которую отдаёт GitHub Pages.
- **Один Next.js-проект**, три зоны через маршруты:
  - `/` — лендинг-развилка (выбор роли + переключатель языка);
  - `/admin/*` — Admin CRM;
  - `/portal/*` — Donor Portal.
- `basePath` и `assetPrefix` = `/CRM_FORT` (имя репозитория), чтобы ассеты
  корректно грузились на `https://0nik5.github.io/CRM_FORT/`.
- `images.unoptimized = true` (обязательно для static export).
- Данные — статические моки с **фиксированным сидом**, чтобы числа не менялись
  между перерисовками. Никаких сетевых запросов в рантайме.

### Причина выбора
Next.js static export ближе к целевому продовому стеку из концепта
(раздел 4: Next.js + Tailwind), при этом даёт чистую статику под Pages.
Компоненты и разметку позже можно переиспользовать при переходе на реальный бэк.

## 3. Инвентарь страниц

### 3.1 Лендинг `/`
- Логотип/название «Фортітьюд ЮА», короткий заголовок-подводка.
- Две крупные карточки-развилки:
  - «Команда фонду» → `/admin`
  - «Кабінет донора» → `/portal`
- Переключатель языка **UA / EN** в углу.

### 3.2 Admin CRM `/admin`
Общий layout: боковое меню + топбар (имя «пользователя»-заглушки, переключатель языка).

1. **Dashboard** `/admin` — строка KPI-карточек + виджеты из концепта (раздел 3.2):
   Grant Pipeline (воронка по стадиям), Cash Flow Forecast (планируемые платежи
   по кварталам), Budget Utilization (план vs факт по активным грантам),
   Donor Concentration (доля доноров), Reporting Calendar (ближайшие дедлайны),
   Compliance Alerts (документы с истекающим сроком).
2. **Донори** `/admin/donors` — таблица (legal_name, type, country, status) +
   карточка донора `/admin/donors/[id]` (реквизиты, контакты, due diligence).
3. **Гранти** `/admin/grants` — таблица-воронка по стадиям (lead…renewed) +
   карточка гранта `/admin/grants/[id]` (транши, бюджет план/факт, отчёты, KPI).
4. **Платежі** `/admin/payments` — список траншей (дата, сумма, транш №, статус
   planned/received/confirmed, bank_statement_ref).
5. **Бюджети** `/admin/budget` — статьи (паливо, зарплата, медикаменти, оренда,
   логістика…), planned vs actual, variance %, флаг отклонения.
6. **Звіти** `/admin/reports` — реестр отчётов (type, deadline, status) + отдельный
   блок державної звітності (ДПС, форми №1-м/№2-м/4ДФ).
7. **Комплаєнс** `/admin/compliance` — санкционные проверки, PEP, сроки документов,
   алерты по истекающим срокам.
8. **Завдання** `/admin/tasks` — список задач/дедлайнов (тип, ответственный,
   дедлайн, статус, привязка к донору/гранту).

### 3.3 Donor Portal `/portal`
Свой лёгкий layout в тёплом «фондовом» бренде (см. раздел 5).

1. **Dashboard донора** `/portal` — карточки виділено / освоєно / залишок +
   таймлайн выплат со статусами.
2. **Бюджет** `/portal/budget` — разбивка по статьям (план vs факт), агрегировано.
3. **Метрики впливу** `/portal/impact` — пациенты / консультации / локации с датами
   (карточки-счётчики + простой график динамики).
4. **Звіти** `/portal/reports` — библиотека отчётов (PDF), кнопки-заглушки
   «Завантажити» (без реального файла).
5. **Апдейти з поля** `/portal/updates` — лента «фото + короткий текст».
   **Опционально** — реализуется, если останется время; иначе выносится в бэклог.

## 4. Данные и языки

- **Моки**: модуль `mock/` (или `lib/mock/`) с массивами по сущностям концепта
  (донори, контакти, гранти, платежі, бюджетні статті, звіти, метрики, комплаєнс,
  завдання, держзвітність). Значения правдоподобные: суммы в грн/USD, украинские
  названия статей и статусов. Фиксированный сид генерации.
- **Графики**: лёгкая библиотека **Recharts** (воронка/бары/линии/пироги).
- **i18n**: простой словарь `{ ua: {...}, en: {...} }` + переключатель. Состояние
  языка хранится в `localStorage` (и/или `?lang=`), по умолчанию UA. Без тяжёлых
  i18n-фреймворков. Текстовые значения данных (статусы, категории) двуязычны;
  имена, суммы, даты — общие для обоих языков.

## 5. Визуальный стиль

Основан на реальном бренде сайта fortitude.org.ua (считан с боевого сайта).

**Палитра:**
- Акцент/бренд: **`#FF6900`** (яркий оранжевый) — кнопки, заголовки, активные
  состояния, акценты на графиках.
- Фон: белый `#FFFFFF` и тёплый off-white `#F9F8F8`; светло-серый `#EBEBEB`
  для разделителей, «заглушённых» блоков, карты.
- Текст: чёрный `#000000` / near-black для body.

**Типографика:**
- Заголовки — **Inter Tight** (heavy/bold, крупные).
- Body/интерфейс — **Inter**.
- Оба шрифта — Google Fonts; при static export подключаются через
  `next/font/google` (self-hosted, без внешних запросов в рантайме).

**Логотип/мотив:**
- Вордмарк «Fortitude» + фирменный знак: сердце с вырезанным медицинским крестом
  (плюсом), оранжевый. Воссоздаётся как SVG.
- Фирменный визуальный ход — **карта Украины в светло-сером с оранжевыми
  «сердечками-пинами» и числами**. Переиспользуется на страницах локаций/метрик
  влияния (Admin `program_data`, Portal «Метрики впливу»).

**Тон по зонам:**
- Admin — деловой, плотный (таблицы, карточки, сайдбар), но в фирменной палитре.
- Portal — воздушный, крупные оранжевые цифры, метрики влияния, «человечный».
- Светлая тема, консистентные отступы и карточки.

## 6. Деплой (GitHub Actions → GitHub Pages)

- Workflow `.github/workflows/deploy.yml`: на push в `master` →
  `npm ci` → `npm run build` (Next static export в `out/`) →
  publish `out/` в GitHub Pages через официальные actions
  (`actions/configure-pages`, `actions/upload-pages-artifact`, `actions/deploy-pages`).
- В репозитории включить Pages в режиме **GitHub Actions**.
- Добавить `.nojekyll` в `out/` (чтобы Pages не резал папки с `_`).
- Итог — публичная ссылка `https://0nik5.github.io/CRM_FORT/`.

## 7. Критерии готовности концепта

- Лендинг с развилкой ролей и переключателем UA/EN работает.
- Все страницы Admin CRM из 3.2 (пп. 1–8) открываются и заполнены мок-данными,
  переходы между списком и карточкой работают.
- Все страницы Donor Portal из 3.3 (пп. 1–4, п. 5 опционально) открываются
  и заполнены мок-данными.
- Переключение языка меняет подписи интерфейса.
- Проект собирается `npm run build` без ошибок и публикуется на GitHub Pages;
  публичная ссылка открывается и кликабельна.
